# EPT - Extended ProtoThreads

EPT - это небольшое расширение над классическими [протопотоками](https://dunkels.com/adam/pt/index.html).

## Простейшее использование
EPT представляет собой STB-библиотеку из одного заголовочного файла. Для использования EPT необходимо включить этот файл в проект и в одном месте определить макрос `#define EPT_IMPLEMENTATION`(обычно в main.c). Также необходимо определить частоту работы вызова прерывания системного таймера `EPT_TICK_FREQ_HZ` в настройках препроцессора.  Затем необходимо поместить в обработчик прерывания системного таймера вызов функции `etp_tick()`.

## Таймеры
Каждый EPT-поток теперь оснащен таймером, позволяющим упростить реализацию не-блокирующего ожидания потока (новый макрос `EPT_SLEEP(ept, time_ms)`)

## Планировшик
EPT включает в себя примитивный round-robin планировщик, автоматизирующий вызов потоков, смену их контекста и переход в энергосберегающий режим (WFI). Для использования планировщика необходимо добавить в проект файл `ept_scheduler.c` и добавить и заполнить `ept_cfg.h`. В `ept_cfg.h` необходимо вручную заполнить массив `threadlist`, функции из которого и представляют из себя потоки, вызываемые планировщиком. После этого необходимо создать саму реализацию функции-протопотока и, если требуется, определить тип ее контекста и создать экземпляр контекста(указатель на который также должен быть занесен в массив `threadlist`). 

## Demo
В директории Demo можно найти простейший пример использования планировщика EPT. В этом примере создается два протопотока, мигающих двумя разными светодиодами (используется плата stm32f030 discovery). 

```c
typedef struct
{
  const pin_t * pin;
  uint32_t period;
}blinker_ctx;

PT_THREAD(blinker(struct ept * ept))
{
    blinker_ctx * ctx = thread_ctx; 
    PT_BEGIN(ept);
    while(1)
    {
        EPT_SLEEP(ept, ctx->period);
        pin_toggle(ctx->pin);
    }
    PT_END(ept);
}
```

Затем информация об этих протопотоках добавляется в массив `threadlist` в файле `ept_cfg.h`:

```C
#include "blinker.h"

extern blinker_ctx blctx1, blctx2; //Conetxts for threads.

const static thread_record threadlist[] = 
{
  {.name = "bl1", .thread = blinker,  .context = &blctx1, .init_state = RUN, .init_dbg_level = OFF}, 
  {.name = "bl2", .thread = blinker,  .context = &blctx2, .init_state = RUN, .init_dbg_level = OFF}
};
```

Также нужно где-то создать сами `blctx1`, `blctx2`. Например, в `main.c`:

```c
#include "main.h"
#define RESOURCES_IMPLEMENTATION
#include "resources.h"
#include "blinker.h"
#include "ept_scheduler.h"
#define EPT_IMPLEMENTATION
#include "ept.h"

blinker_ctx blctx1 = 
{
  .pin = &C9_pin, 
  .period = 250
};

blinker_ctx blctx2 = 
{
  .pin = &C8_pin, 
  .period = 400
};

int main(void)
{
  hw_init();
  ept_scheduler();
}
```
В самой функции `int main` происходит только первичная инициализация периферии (часто при помощи stm32cube), после чего управление передается планировщику и никогда не возвращается назад. 

# EPT CLI

В составе EPT есть реализация командного интерфейса, работающего через USART. Практически в каждом проекте необходим командный интерфейс для отладки и тестирования. В минимальной реализации командный интерфейс позволяет управлять аппаратными ресурсами(например, GPIO), самими протопотоками, а также получать информацию об устройстве. Расширение функционала CLI ложится на плечи программиста и зависит от нужд конкретного проекта.  

## Профилировщик

В файле `ept_cfg.h` находится определение макроса PROFILER, который позволяет собрать проект с включенным профилировщиком задач. Это добавляет небольшие накладные расходы в основной цикл планировщика, поэтому может быть целесообразно отключать профилировщик в релизной версии ПО. Хотя сам профилировщик работает независимо, для того, чтобы получать результаты его работы, требуется CLI. При каждом вызове команда `prof` выдает наибольшее время исполнения каждого потока (время, на которое управление передавалось функции) с момента прошлого использования команды `prof`.
