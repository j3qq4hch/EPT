# EPT - Extended ProtoThreads

EPT - это самостоятельная реализация кооперативных потоков, выросшая из классических [протопотоков](https://dunkels.com/adam/pt/index.html) Адама Данкелса. От исходных pt осталась только идея local continuations на `switch` (`lc.h`); слой `pt.h` упразднён - всё ядро живёт в одном заголовке `ept.h`.

## Простейшее использование
EPT представляет собой STB-библиотеку из одного заголовочного файла. Для использования EPT необходимо включить `ept.h` в проект и в одном месте определить макрос `#define EPT_IMPLEMENTATION` (обычно это делает `ept_scheduler.c`). Также необходимо определить частоту вызова прерывания системного таймера `EPT_TICK_FREQ_HZ` в настройках препроцессора. Затем необходимо поместить в обработчик прерывания системного таймера вызов `ept_scheduler_tick()` (или `ept_tick()`, если планировщик не используется).

## Статусы потоков
Каждый вызов потока возвращает один из трёх статусов:

| Статус | Значение |
|---|---|
| `EPT_ACTIVE` | у потока есть работа прямо сейчас - засыпать нельзя |
| `EPT_BLOCKED` | поток ждёт условия или таймера - MCU может спать до следующего тика/прерывания |
| `EPT_DONE` | поток завершился (имеет смысл для дочерних потоков) |

Инвариант: `EPT_ALIVE(s) == (s < EPT_DONE)`.

`EPT_SPAWN` пробрасывает статус дочернего потока родителю без изменений, поэтому спящий на любом уровне вложенности поток не мешает переходу MCU в энергосберегающий режим.

## Основные макросы
- `EPT_THREAD(name(struct ept *ept))` / `EPT_BEGIN(ept)` / `EPT_END(ept)` - определение потока;
- `EPT_WAIT_UNTIL / EPT_WAIT_WHILE` - блокирующее ожидание условия;
- `EPT_WAIT_UNTIL_TIMEOUT` - ожидание условия с таймаутом;
- `EPT_SLEEP(ept, time_ms)` - неблокирующий сон на встроенном таймере потока;
- `EPT_YIELD` - отдать один проход планировщика, продолжить;
- `EPT_YIELD_UNTIL / EPT_YIELD_WHILE` - уступить минимум один раз, затем ждать условия;
- `EPT_SPAWN / EPT_WAIT_THREAD` - запуск дочернего потока с пропагацией статуса;
- `EPT_RESTART / EPT_EXIT` - перезапуск/досрочное завершение.

Ограничения switch-based LC: внутри тела потока нельзя использовать `switch` через точку блокировки, два блокирующих макроса не должны находиться на одной строке, локальные переменные не переживают блокировку (используйте `static` или контекст).

## Планировщик
EPT включает в себя примитивный round-robin планировщик, автоматизирующий вызов потоков, смену их контекста и переход в энергосберегающий режим. Для использования планировщика необходимо добавить в проект файл `ept_scheduler.c` и заполнить `ept_cfg.h` (шаблон - `ept_cfg_template.h`): массив `threadlist` перечисляет потоки верхнего уровня.

Энергосбережение (`#define LOW_POWER_MODE` в `ept_cfg.h`): если за проход ни один поток не вернул `EPT_ACTIVE`, планировщик вызывает `drv_sleep()` (WFI из `drv_basic_*.h`). Любое прерывание, включая SysTick, будит MCU, поэтому сон ограничен одним тиком и опрашиваемые условия не могут быть пропущены.

```c
typedef struct
{
  const pin_t * pin;
  uint32_t period;
} blinker_ctx;

EPT_THREAD(blinker(struct ept * ept))
{
    blinker_ctx * ctx = thread_ctx;
    EPT_BEGIN(ept);
    while(1)
    {
        EPT_SLEEP(ept, ctx->period);
        pin_toggle(ctx->pin);
    }
    EPT_END(ept);
}
```

Информация о потоках добавляется в массив `threadlist` в файле `ept_cfg.h`:

```c
#include "blinker.h"

extern blinker_ctx blctx1, blctx2;

const static thread_record threadlist[] =
{
  {.name = "bl1", .thread = blinker, .context = &blctx1, .init_state = RUN, .init_dbg_level = OFF},
  {.name = "bl2", .thread = blinker, .context = &blctx2, .init_state = RUN, .init_dbg_level = OFF}
};
```

В `int main` происходит только первичная инициализация периферии, после чего управление передаётся `ept_scheduler()` и никогда не возвращается назад.

# EPT CLI

В составе EPT есть реализация командного интерфейса, работающего через USART. Практически в каждом проекте необходим командный интерфейс для отладки и тестирования. В минимальной реализации командный интерфейс позволяет управлять аппаратными ресурсами (например, GPIO), самими протопотоками, а также получать информацию об устройстве. Расширение функционала CLI ложится на плечи программиста и зависит от нужд конкретного проекта.

## Профилировщик

В файле `ept_cfg.h` находится определение макроса PROFILER, который позволяет собрать проект с включенным профилировщиком задач. Это добавляет небольшие накладные расходы в основной цикл планировщика, поэтому может быть целесообразно отключать профилировщик в релизной версии ПО. Хотя сам профилировщик работает независимо, для того, чтобы получать результаты его работы, требуется CLI. При каждом вызове команда `prof` выдает наибольшее время исполнения каждого потока (время, на которое управление передавалось функцией) с момента прошлого использования команды `prof`.
